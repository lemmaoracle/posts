# ユースケース: 金融データ流出防止

## 問題

従業員が組織間で出向する場合（例: 保険会社 → 銀行代理店）、顧客データに対する信頼境界があいまいになります。出向中のスタッフは元の組織と先の組織の両方のシステムにアクセス権を保持するため、不正なデータ流出の窓口が生じます。

**引き金となった事例:** MetLife Japan (2026年) — 出向中の従業員が36の代理店パートナー（広島銀行、福岡銀行など）から顧客データを流出させ、2,476件の記録に影響を与えました。日本の生命保険史上、最大の事例です。上位4社の生命保険会社全体で、約3,500件の事例が確認されました。金融庁は保険業法に基づく報告命令を発出しました。

## Lemmaの価値提案

| 機能 | Lemmaの役割 |
|---|---|
| アクセス証跡（誰が何に、いつアクセスしたか） | ✅ 中核 — プライバシー保護を伴うZK証明によるアクセスログ |
| リアルタイム流出検知 | ⚠️ 補完 — DLP/SIEMと統合; Lemmaは証明レイヤーを提供 |
| 事後の改ざん証拠 | ✅ 中核 — 改ざん不可能で検証可能な証跡 |
| 心理的抑止力 | ✅ 間接的 — 「否認できない証拠が存在する」ことが行動を変える |

### 主要なポジショニング

**検知 = 既存ツール (DLP, SIEM)。 証明 = Lemma。**

LemmaはDLP/SIEMと競合せず、それらを完結させます。DLPは異常をフラグ付けし、Lemmaはアクセスが発生したこととログが改ざんされていないことを証明します。これは防御可能で重複しないポジションです。
