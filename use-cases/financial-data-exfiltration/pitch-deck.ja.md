# ピッチデック: 金融データ流出 — 1枚スライド

---

## 2,476件の記録が盗まれた。証明ゼロ。

**問題:** 出向従業員が顧客データを持ち去るとき、何が起きたかを証明できません。あなたのDLPは異常をフラグ付けします。ログは彼らがレコードにアクセスしたと言います。しかしあなたのログは変更可能で、分断されており、争う余地があります。

**規制当局はあなたのスプレッドシートを求めていません。彼らは証拠を求めています。**

### Lemma導入前

| ステップ | 発生すること | ギャップ |
|---|---|---|
| 従業員が出向 | 両組織の認証情報 | 信頼境界の強制なし |
| データアクセス | 変更可能なDBに保存されたログ | ログは改ざん可能 |
| 流出検知 | DLPが異常をフラグ付け | 「何かが起きた」 — 何が*起きたか*ではない |
| 監査要求 | 2週間の手動ログ・フォレンジクス | 組織間の検証可能性なし |

### Lemma導入後

| ステップ | 発生すること | 利点 |
|---|---|---|
| 従業員が出向 | Lemmaが時間制限付きアクセス認証情報を発行 | 信頼境界が明示的 |
| データアクセス | アクセスごとにZK認証生成 | 暗号学的証明、プライバシー保護 |
| 流出検知 | DLP + Lemma = 異常 + 証明 | 数分で否認不可能な証拠 |
| 監査要求 | 検証可能な認証セットをエクスポート | 数週間ではなく数時間 |

### スタック: 補完的であり、競合的ではない

```
┌──────────────────────────────────┐
│  検知  │  DLP / SIEM / CASB │  ← 「何かが起きた」
├─────────────┼────────────────────┤
│  証明      │  Lemma Oracle      │  ← 「これが暗号学的証明です」
└─────────────┴────────────────────────┘
```

**あなたの既存ツールは検知します。Lemmaは証明します。**
