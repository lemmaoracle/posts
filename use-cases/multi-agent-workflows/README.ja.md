# ユースケース: マルチエージェント・ワークフロー

## テーゼ

**委任した ≠ 追跡できる**

オーケストレータがサブエージェントに業務を割り振る。サブエージェントはMCPサーバーを呼び、ツールを起動し、結果を返す。来歴レイヤがなければ、鎖は不透明になる──誰が誰に許可したか、誰が何を見たか、誰が何を決めたか。Lemmaは委任の各ノードに証明を付ける。最終成果物には、最初の発行者まで遡れる検証可能な経路が残る。

## Lemmaが証明するもの

- 委任の連鎖（主体 → 受任者 → 再委任先）
- 各ステップで継承されるスコープ
- ノードごとの入力と判断
- エンドツーエンドの監査可能性

## 問題

現代のAIワークフローは本質的にマルチエージェントである：オーケストレータがタスクを分解し、専門家に委任し、結果を集約する。各専門家はMCP経由で外部ツールを呼び出し、APIを叩き、サブエージェントを生成する可能性がある。

問題はトレーサビリティだ。何かが起こったとき——誤った判断、データ漏洩、不正アクセス——監査証跡はログ、エージェントメモリ、ツールレスポンスにまたがって断片化している。いかなる当事者も完全な委任の連鎖を再構築できない：

- **誰が**専門家エージェントにそのデータへのアクセスを許可したか？
- **何の**スコープが各ステップで委任されたか？
- **どの**入力が最終的な決定に影響したか？
- **どうやって**権限は元の主体から伝播したか？

現在のアプローチはロギングとモニタリングに依存している——事後的な再構築であり、暗号論理的証明ではない。ログは変更可能で、不完全で、システム間に分散している。

## Lemmaがどう変えるか

Lemmaはマルチエージェントワークフローに**証明可能な委任の連鎖**を提供する：

1. **委任時:** 各委任ステップが、委任者・受任者・スコープ・タイムスタンプを束縛するZK証明を生成する。証明はオンチェーンにアンカーされる。
2. **実行時:** 各エージェント操作がその委任証明を携える。下流のツールとAPIは、エージェントの自己アテステーションを信頼することなく、呼び出し元の権限を検証できる。
3. **集約時:** オーケストレータは、元の主体からすべての委任ノードを経て最終結果に至る完全な証明チェーンとともに最終成果物を組み立てる。

委任の連鎖が監査証跡であり、証明が証拠であり、オンチェーンのアンカーが事後的に何も改ざんされていないことを保証する。
