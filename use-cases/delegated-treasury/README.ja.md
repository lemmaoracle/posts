# ユースケース: Delegated Treasury

## テーゼ

**承認されている ≠ 証明されている**

企業がAIエージェントに予算と業務範囲を渡す。月10万円まで、SaaSサブスクのみ、5万円超は要承認──この制約はソフトプロンプトに置けない。Lemmaは、発行組織の署名付きで、これらの制約をオンチェーンの支出制御アテステーションとして発行する。決済の相手側はそのアテステーションを検証してから支払いを受け入れる。

## Lemmaが証明するもの

- 発行主体（組織・部門）
- 支出上限とスコープ
- 有効期間
- 失効ステータス

## 問題

企業がAIエージェントに支出権限を委譲するとき、制約は典型的に次のいずれかで実装される：

1. **ソフトプロンプト** — 「$500以上使わないで」とエージェントに指示する。これは提案であって保証ではない。プロンプトインジェクション、コンテキストオーバーフロー、または敵対的交渉によって無効化される。

2. **プラットフォームガードレール** — エージェントプラットフォームが制限を強制する。しかし、これらは中央集権的で不透明、かつポータブルではない。売り手は、プラットフォームを信頼することなく、買い手の権限を独立して検証する手段を持たない。

どちらのアプローチも、エージェントの支出権限が本物で、現在有効で、制約付きであることの暗号論理的証明を提供しない。その証明がなければ、売り手はエージェントの主張を信じるか、取引を拒否するしかない──どちらも最適ではない。

## Lemmaがどう変えるか

Lemmaは委譲エージェントに**検証可能な支出権限**を提供する：

1. **委譲時:** 発行組織がオンチェーンの支出制御アテステーションを作成する：主体の本人性、支出上限、カテゴリスコープ、有効期間、失効エンドポイント。
2. **取引時:** 売り手エージェントは支払いを受け入れる前にアテステーションを検証する。アテステーションは発行組織によって署名されている——エージェントでもプラットフォームでもない。
3. **監査時:** すべての取引が、それを認可した委譲の暗号論理的証明を伴う。コンプライアンスチームは検証可能な証拠を得る。ログエントリではない。

委譲が証明であり、アテステーションが契約であり、オンチェーンのアンカーが保証である。
