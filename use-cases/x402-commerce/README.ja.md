# ユースケース: X402 Commerce

## テーゼ

**決済される ≠ 検証されている**

x402のレール上で、買い手エージェントと売り手エージェントが初めて出会い、ミリ秒単位でmicropaymentを実行する。Lemmaは、売り手が主張する属性──価格、在庫、SLA、本人性──をそれぞれZK証明で包む。買い手は平文の主張を信じない。支払う前に、暗号的に検証する。

## Lemmaが証明するもの

- 売り手の本人性と権限
- 提示された在庫と価格
- サービスレベルの確約
- 発行者署名の来歴

## 問題

エージェント間コマースでは、取引はミリ秒で決済されるが、検証は不在かレピュテーションシステムに委ねられている。売り手エージェントは「在庫がある、価格はX、SLA Y以内で納品する」と主張する。買い手エージェントは支払う。その主張が虚偽だった場合、支払いは既に完了している。

レピュテーションシステムは事後にペナルティを与えるだけ。コミットを防ぐことはない。そして、1回の取引のために立ち上げられ、すぐに破棄される一時的なエージェントの世界では、レピュテーションは無意味だ。取引した売り手エージェントは、嘘が発覚する頃には存在していないかもしれない。

## Lemmaがどう変えるか

Lemmaはエージェントコマースに**取引前検証**を提供する：

1. **出品時:** 売り手エージェントの属性（本人性、権限、在庫、価格、SLA）が、検証可能な権限によるZK証明に包まれる。
2. **交渉時:** 買い手エージェントは取引をコミットする前に証明を検証する。証明なき取引は成立しない。
3. **決済時:** 双方が、何が主張され何が検証されたかの暗号的証拠を保持する。

x402プロトコルは決済レールを担う。Lemmaは信頼レールを担う。
