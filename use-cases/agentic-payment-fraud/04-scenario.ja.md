---
title: "実例：通ってはいけなかった支払い"
---


2026 年 4 月、中堅製造業の調達チームが、定期 SaaS 請求の決済を AI エージェントに任せる運用を始めました。エージェントには決済プロバイダの API キーと、月額上限 USD 50,000 のプロンプト側ガードレールが与えられています。2 ヶ月後、社内監査が誰も把握していない取引相手への USD 180,000 の支払いを掘り起こします。事業を行っていない jurisdiction の相手でした。

事後再構成は困難です。API キーは有効でした。プロバイダのログにはエージェントが呼び出した記録が残っています。プロンプト側ガードレールは、上流ツール応答に紛れた異常な指示で迂回されていました。ログだけからは「エージェントがこの支払いをこの相手に通す権限を本当に持っていたか」を証明できません。

Trust402 が決済レールの前段にあれば、コールごとにアテステーションが添付されています。委任した principal、当該期間の spend cap、相手側が満たすべき jurisdiction 属性。上限超過の支払いは検証可能な証明を生成できず、決済コントラクトが確定前に拒否します。プロンプトログを事後に読み解く必要はありません。

業種別の委任パターン（財務オペ・マルチエージェント決済・API 課金）、x402 / MCP / A2A 統合スケッチ、Base Sepolia 上の Reference implementation のウォークスルーは、相談コール後の業種別資料でご共有しています。
