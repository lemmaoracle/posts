# ユースケース: API 課金を、AI エージェントに任せる。

## テーゼ

**API キーを渡す ≠ 範囲を渡す。**

AI エージェントに外部 API を呼ばせたい。だが API キーをそのまま渡せば、課金上限も呼び先も期限も縛れません。「月 $100 まで」「この endpoint だけ」「今月末まで」という範囲を、検証可能な委任として固定する必要があります。

## Lemma が証明するもの

- 委任元（組織・開発者）と呼び出しエージェント
- 課金できる上限額（spendLimitUSDC）
- 許可された API endpoint の範囲（scope）
- 有効期限と失効ステータス

## 問題

エージェントに API を呼ばせるとき、権限の制約は典型的に次のいずれかで実装されます。

1. **API キーをそのまま委譲** — キーが漏れれば、上限も呼び先も無制限。漏洩は構造的に防げません。
2. **アプリ側のレート制限** — 自前のラッパーで上限を効かせる。だが呼び先や課金の正当性を、第三者が独立に検証する手段がありません。

どちらも、エージェントの API 呼び出しが「正規の委任の範囲内だった」ことの証明を残しません。

## Lemma がどう変えるか

Lemma は呼び出しエージェントに**範囲付きの検証可能な権限**を与えます。

1. **委任時:** 組織・開発者が、課金上限・許可 API・有効期限を持つ委任証明を発行します。API キーそのものはエージェントに渡しません。
2. **呼び出し時:** x402 middleware が API call の前に Trust402 で権限を確認します。範囲内なら自律実行、範囲外は実行前に止まります。
3. **監査時:** すべての呼び出しに、それを認可した委任の証明が付きます。キーや課金権限の中身を開示せずに、「範囲内だった」ことを示せます。

委任が証明であり、範囲が契約です。API キー漏洩のリスクは、構造的に解消されます。
