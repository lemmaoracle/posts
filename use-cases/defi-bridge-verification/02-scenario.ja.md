# シナリオ: Kelp DAO / rsETHブリッジ悪用

## Lemma導入前 — どのように攻撃が展開されたか

### 背景

Kelp DAOはEigenLayer上に構築された流動性リステーキングプロトコルです。その旗艦トークン、rsETHは、預け入れ、ステーキングされ、リステーキングされたETHを表します。rsETHは、LayerZero OFT（Omnichain Fungible Token）アダプターを介して20以上のチェーンに展開され、攻撃時点で約15.7億ドルのTVLがありました。

### 攻撃タイムライン (2026年4月18–19日)

| 時間 (UTC) | イベント | 詳細 |
|---|---|---|
| ~07:35 (T-10h) | 攻撃者の準備 | Tornado Cashで資金を供給した9つの運用ウォレット (1 ETH混合プール、ガス代としてそれぞれ約0.0978 ETH) |
| 17:35 | 攻撃実行 | 攻撃者がスプーフィングされたクロスチェーンメッセージでLayerZeroのEndpointV2コントラクトの`lzReceive`を呼び出し |
| 17:35 | DVN検証 | 1-of-1 DVN検証者が偽メッセージを承認 — 1つの署名のみが必要 |
| 17:35 | 資金放出 | KelpのOFTアダプターが116,500 rsETH (約2億9200万ドル) を攻撃者制御アドレスに放出 |
| 17:35–18:20 | 担保変換 | 攻撃者がrsETHをAave V3に担保として預け入れ、それに対して約106,467 WETHを借り入れる |
| 18:21 | 緊急停止 | Kelpの緊急マルチシグが`pauseAll`を実行 — 最初の悪意あるトランザクションから46分後 |
| 18:26, 18:28 | 追跡試行ブロック | 同じLayerZeroパケットが別の約40,000 rsETH (約1億ドル) を狙う — 停止が保持 |
| ~18:00–20:00 | 連鎖的な凍結 | Aave, SparkLend, FluidがrsETH市場を凍結。Ethenaは予防措置としてLayerZero OFTブリッジを一時停止 |
| ~20:10 | 最初の公表 | Kelp DAOがXに投稿 — 資金流出から約3時間後 |
| 攻撃後 | ログ削除 | 侵害されたRPCノード上のマルウェアが自身とローカルログファイルを削除 |

### なぜ成功したか: 3つの構造的失敗

1. **1-of-1 DVN設定** — Kelp DAOは約3億ドルの担保を保持するブリッジのセキュリティ全体を単一の署名鍵に委任しました。しきい値なし、冗長性なし、フォールバックなし。

2. **検証なしのRPC信頼** — 検証者は真実のソースとしてRPCノードを信頼しました。攻撃者がKelpのRPCノードのうち2つを侵害し、その後残りの正直なノードをDDoS攻撃してフェイルオーバーを強制し、攻撃者制御ノードに切り替えさせました。検証者はクロスチェーンメッセージが実際に主張されたソースチェーンから発信されたことを独立して検証する方法を持ちませんでした。

3. **実行前の起点チェックなし** — OFTアダプターは完全に証明されたクロスチェーンメッセージとして見えるものを受け取り、設計通りに実行しました。以下の問いをするメカニズムがありませんでした: 「この状態変更は、発行時に検証可能な条件下で、主張するエンティティによって、主張するソースチェーン上で実際に発行されたのか？」

### 連鎖的影響

| 影響 | 規模 |
|---|---|
| 直接損失 (rsETH流出) | 2億9200万ドル |
| Aave V3不良債務 (清算不能なrsETH担保) | 約1億7700万ドル |
| エコシステムTVL流出 (攻撃後2日間) | 130億ドル以上 |
| LRTトークンのペグ喪失 | rsETHが20以上のチェーンでペグを失う |
| プロトコル凍結 | Aave V3/V4, SparkLend, Fluid, Ethena |
| ZROトークンへの影響 | 24時間で-22% |

## Lemma導入後 — 同じ攻撃がどのように防がれるか

### 導入時の変化

1. **発行側の起点証明** — ソースチェーン上でクロスチェーンメッセージが生成されるとき、Lemmaはアクションをそのソースに紐づけるZK起点証明を生成します: 発行エンティティ、起点チェーン、発行時の条件。

2. **証明はメッセージと共に移動** — 起点証明は転送中にクロスチェーンメッセージに添付されます。送信側の継続的な参加や誠実さに依存しません。

3. **実行前検証** — 受信システム（OFTアダプター）がコミットする前に、Lemmaの実行前認証を実行します: 起点証明を検証します。アクションは証明が検証された場合のみコミットされます。そうでなければ書き込み時に拒否されます。

### 攻撃タイムライン（Lemma導入時）

| 時間 (UTC) | イベント | Lemmaによる変化 |
|---|---|---|
| ~07:35 | 攻撃者の準備 | 同じ |
| 17:35 | 攻撃者がスプーフィングされたメッセージで`lzReceive`を呼び出し | 同じ |
| 17:35 | DVN検証 (依然として1-of-1) | DVNが承認 (依然として侵害済み) |
| 17:35 | **実行前認証が実行** | **Lemma検証が偽メッセージを拒否 — 起点証明が検証されない。メッセージは主張されたソースチェーン上で検証可能な条件下で発行されていない。** |
| 17:35 | **資金放出: 拒否** | **OFTアダプターはコミットしません。116,500 rsETHはエスクローに留まります。** |
| 17:35–18:20 | 担保変換は不可能 | 攻撃は境界で停止。rsETH放出なし → Aave預け入れなし → 不良債務なし |
| 18:21+ | オプション: アラート発動 | Lemmaの検証失敗はプロトコルの監視スタックへのアラートを引き起こす可能性あり |

### これが構造的に意味すること

| 質問 | Lemmaなし | Lemmaあり |
|---|---|---|
| 1-of-1 DVNは悪用可能か？ | はい — 単一障害点 | 依然として可能ですが、DVN承認に関わらず偽メッセージはコミット前に拒否される |
| 侵害されたRPCノードは偽メッセージを注入可能か？ | はい — 検証者はRPCを信頼 | いいえ — 起点証明はRPCの信頼性と独立して検証される |
| ログ削除で痕跡を消せるか？ | はい — マルウェアがローカルログを削除 | 部分的に — オンチェーン固定された認証は生き残るが、リアルタイムアラートが主要防御 |
| コミットの瞬間に何が起きるか？ | 受信システムはDVN署名のみに基づいてコミット | 受信システムは起点証明 + DVN署名の両方が検証された場合のみコミット |
| 悪用の停止速度は？ | 46分 (手動緊急停止) | 瞬時 (書き込み時の自動拒否) |

### 重要な洞察: 多層防御

LemmaはDVNレイヤーを置き換えません。第2の独立した検証レイヤーを追加します:

```
現在:   DVN署名  ──────────────────▶  コミット
                                    ↑ 単一の信頼点

Lemma導入: DVN署名  ──┐
          起点証明   ──┼──────────────▶  コミット
                                ↑ 2つの独立した
                                  検証が要求される
```

DVNが侵害された場合でも（1-of-1、または不誠実な多数派を持つN-of-M）、起点証明も検証されなければなりません。これは多層防御です — 2つのレイヤーは独立して失敗します。
