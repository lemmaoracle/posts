# クロスケース: TradFi（MetLife）× DeFi（Kelp DAO）— 統一証明レイヤー

## 統一メッセージ

**TradFiもDeFiも同じ構造的ギャップを共有しています：受信システムが出所を検証せずにコミットしている。**

表面は異なります — 従業員によるデータ持ち出し vs. ブリッジエクスプロイト — しかし障害モードは同一です：暗号論理的に有効で意味的に誤ったアクションが、独立した検証が存在しないためにコミットされること。

## 横並び比較

| 側面 | MetLife（TradFi） | Kelp DAO（DeFi） |
|---|---|---|
| **損失** | 2,476件の顧客記録 | $292M |
| **表面** | 出向社員が組織間境界を越えてデータにアクセス | 攻撃者がチェーン間境界を越えてメッセージを偽装 |
| **根本原因** | 信頼境界の曖昧さ（組織間データアクセスポリシーなし） | 信頼境界での出所の前提（1-of-1 DVN） |
| **「有効」だったもの** | 従業員は両組織の正当な認証情報を持っていた | クロスチェーンメッセージは有効なDVN署名を持っていた |
| **誤っていたもの** | アクセスは暗黙のデータガバナンス規則に違反 | メッセージは主張された送信元チェーンから発信されていなかった |
| **事後対応** | 数週間のログフォレンジック、相互非難 | 46分の一時停止、$177Mの不良債権、$13BのTVL流出 |
| **欠けていたもの** | 誰がいつ何にアクセスしたかの暗号論理的証明 | メッセージの出所が本物であることの事前実行検証 |
| **Lemmaの修正** | データアクセスのZKアテステーション（改ざん検知可能な監査証跡） | クロスチェーンメッセージのZK出所証明（事前実行アテステーション） |

## パターン: 「有効だが誤り」

```
TradFi:   従業員認証情報は有効 + アクセスは意味的に不正 = データ持ち出し
DeFi:     DVN署名は有効 + メッセージは意味的に偽造 = $292Mのエクスプロイト
```

どちらのケースでも、既存の検証レイヤーは**暗号論理的ラッパー**（認証情報、署名）を検証するが、**意味的コンテンツ**（認可スコープ、出所の真実性）を検証していない。Lemmaは欠けている意味的検証を提供する。

## 統一アーキテクチャ

```
                    ┌─────────────────────────────┐
                    │      Lemma Trust Layer       │
                    │  （検証可能な出所証明）        │
                    └──────────┬──────────────────┘
                               │
                ┌──────────────┼──────────────┐
                ▼              ▼              ▼
        ┌──────────────┐ ┌──────────┐ ┌──────────────┐
        │   TradFi     │ │  DeFi    │ │  AI / Agent  │
        │  アクセス     │ │  ブリッジ │ │  ツール/API  │
        │  来歴         │ │  出所    │ │  認可        │
        └──────────────┘ └──────────┘ └──────────────┘
```

同じ検証プリミティブ — 「コミット前に出所を証明する」 — が3つのドメインすべてに適用される。回路とポリシーの詳細は異なるが、アーキテクチャパターンは共通：

1. **発行:** 状態変異の瞬間に出所証明を生成
2. **輸送:** 信頼境界を越えてアクションに伴い証明を運ぶ
3. **検証:** 受信システムはコミット前に検証；証明が失敗すれば拒否

## 営業ポジショニング

### TradFiの見込み客向け

> 「従業員が顧客データを持ち出したとき、DLPは異常を検出しますが、何が起きたかを証明できません。規制当局から問われたら、ログフォレンジックに数週間かけます。$292Mのブリッジエクスプロイトを止めるのと同じ証明レイヤーが、あなたの監査証跡を否認不能にします — 数週間ではなく数時間で。」

### DeFiの見込み客向け

> 「MetLifeで従業員が2,476件の記録を持ち出したのと同じ構造的ギャップが、Kelp DAOから$292Mを流出させました。どちらのケースでも、アクションは『有効』でしたが意味的に誤っていました。どちらのケースでも、受信システムは出所を検証せずにコミットしました。Lemmaは境界の両側で機能する証明レイヤーです。」

### クロスドメインの見込み客向け（両方を持つ企業）

> 「コンプライアンスチームは否認不能な監査証跡が必要です。DeFi運用は事前実行アテステーションが必要です。これらは同じプリミティブです：検証可能な出所。一つの証明レイヤー、二つの表面。」

## プロダクトロードマップへの含意

統一パターンは、Lemmaの証明バンドルプリミティブ（Poseidon/BN254、BBS+/BLS12-381、Groth16）をドメイン固有のポリシーレイヤーを持つ**共有検証基盤**として提示すべきことを示唆しています：

| レイヤー | 共通 | ドメイン固有 |
|---|---|---|
| 回路プリミティブ | Poseidon、BBS+、Groth16 | すべて共通 |
| 証明生成 | 発行側の出所証明 | 同一パターン |
| 証明検証 | 事前実行アテステーション | 同一パターン |
| ドメインポリシー | — | TradFi: アクセススコープ、保持ルール; DeFi: リプレイ防止、カストディパス、再質押深度; AI: 認可スコープ、モデルバージョン |

このアーキテクチャにより、単一の営業会話で複数のバーティカルに対応でき、コアを各ドメイン向けに再構築する必要がありません。
