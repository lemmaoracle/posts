# アーキテクチャ: クロスチェーンブリッジ向け実行前認証

## 設計原則

**受信システムはコミット前に起点を検証しなければならない。**

現在のブリッジアーキテクチャは検証の負担を送信側に置いています（DVN署名、マルチシグしきい値）。Lemmaは信頼の座を受信側に移します: 起点が独立して検証されない限りコミットしない。

## 現行アーキテクチャ（簡略化）

```
Source Chain                              Destination Chain
┌──────────────┐    LayerZero OFT     ┌──────────────────┐
│  Kelp DAO    │────────────────────▶│  OFT Adapter     │
│  Contract    │   cross-chain msg   │                  │
└──────────────┘                     │  DVN verify?     │
                                     │  ✅ 1-of-1 sig   │
                                     │       │          │
                                     │       ▼          │
                                     │   COMMIT         │
                                     │  (release funds) │
                                     └──────────────────┘
                                        ↑
                            単一の信頼点:
                            DVN署名のみがアクションが
                            コミットするかどうかを決定
```

**失敗モード:** DVNが侵害された場合、またはRPCがスプーフィングされた場合、受信システムはメッセージを拒否する独立した方法を持ちません。単一の信頼前提に基づいてコミットします。

## コンポーネント詳細

### 1. Lemma起点認証ゲートウェイ（ソース側）

ソースチェーン上の発行境界に配置されます。クロスチェーンメッセージが生成されるとき:

- ZK起点証明を生成: `proof(issuer_id, source_chain_id, action_hash, conditions_hash, timestamp)`
- 証明は機密状態を明かさずにアクションをそのソースに紐づける
- ハッシュ化された条件には以下を含む: 支払能力ステータス、カストディ経路、再担保深さ
- 証明は認証ペイロードとしてクロスチェーンメッセージに添付される

**回路プリミティブ:** Poseidon over BN254（回路内コミットメント）、BBS+ over BLS12-381（発行側選択的開示）、Groth16（ZK証明）

### 2. Lemma実行前検証器（デスティネーション側）

OFTアダプターの受信パスに統合されます。アダプターがコミットする前に:

1. 着信メッセージから起点証明を抽出
2. ZK証明を期待される発行者、ソースチェーン、条件に対して検証
3. ドメインポリシーレイヤーをチェック: 再生防止、カストディ経路検証、再担保深さ制限
4. 検証が成功した場合 → コミットに進む
5. 検証が失敗した場合 → メッセージを拒否、失敗をログ記録、アラート発動

**この検証はDVNレイヤーから独立しています。** DVN、RPCノード、送信側の継続的誠実さを信頼しません。

### 3. ドメインポリシーレイヤー

回路の上に、構成可能なポリシーレイヤーがあり、どの起点形状が受信システムにとって許容可能かを制約します:

| ポリシー | 強制する内容 |
|---|---|
| 再生防止 | 各起点証明は一意のナンスに紐付けられ、再生された証明は拒否される |
| カストディ経路検証 | 証明はブリッジされる資産が預金からブリッジエスクローへの検証可能なカストディ経路を持つことを認証しなければならない |
| 再担保深さ制限 | 証明は原資産の再担保深さが設定された最大値を超えないことを認証しなければならない |
| 支払能力認証 | 証明は発行者が発行時に支払能力があったことを認証しなければなければならない（オンチェーン状態に基づく） |

ポリシーはプロトコルごとに構成可能です。保守的なプロトコルは4つすべてを要求できますし、より寛容なプロトコルはサブセットを有効にできます。

### 4. オンチェーン認証固定

すべての起点証明はMerkleツリーにコミットされ、ルートは定期的にオンチェーンに固定されます。これにより以下が提供されます:

- ログ削除を生き延びる事後フォレンジクス証拠
- 規制コンプライアンスのための監査証跡
- 紛争解決: どの当事者も何がいつ認証されたかを検証可能

## 統合パス

| パス | 説明 | 工数 | 信頼保証 |
|---|---|---|---|
| **OFTアダプターフック** | Lemma検証器をLayerZeroの`lzReceive`パスにpre-commitフックとして統合 | 中程度 | 最強 — いかなる状態変更前の検証 |
| **DVN拡張** | DVNをLemma起点証明で置き換えるか、追加の検証要件として拡張 | 中程度 | 強力 — 既存DVNとの多層防御 |
| **監視オーバーレイ** | Lemmaが既存フローと並行して起点証明を検証; 不整合を手動介入のためにフラグ付け | 低い | 最弱 — 検知のみ、自動拒否なし |

### 推奨: OFTアダプターフック

OFTアダプターフックは推奨統合パスです。なぜなら、**起点検証なしに状態変更が発生しない**ことを保証するからです。これは現在のアーキテクチャが欠いている「コミット前に検証」保証です。

```
lzReceive() {
  1. クロスチェーンメッセージを抽出
  2. DVN検証を実行 (既存)
  3. Lemma実行前認証を実行 (新規)
     └── 失敗した場合: revert + alert
  4. 状態変更をコミット (両方が通過した場合のみ)
}
```

## 失敗注入

[example-origin](https://github.com/lemmaoracle/example-origin) リポジトリには失敗注入モードが含まれています:

- 暗号的に有効だが意味的に間違った遷移（偽のクロスチェーンメッセージ）が注入される
- 検証がコミット前にそれを拒否する
- 悪用が境界で停止され、事後に検知されないことを実証

## セキュリティ考慮事項

- **ゲートウェイバイパス**: ソース側ゲートウェイが迂回された場合（起点証明なしでメッセージが送信）、デスティネーション検証器はそれらを拒否します。これは正しい振る舞いです — デフォルトは受理ではなく拒否です。
- **証明サイズとガスコスト**: Groth16証明は約128バイトです。検証は単一のペアリングチェックです。ガスオーバーヘッドはブリッジトランザクション自体に比べて最小限です。
- **証明の新鮮さ**: 起点証明にはタイムスタンプとナンスが含まれます。古い証明は拒否されます。これにより再生攻撃を防ぎます。
- **固定頻度**: MerkleルートをNブロックまたはT秒ごとに固定（構成可能）。頻度が高いほどフォレンジクス保証が強力 + オンチェーンコストが高い。
