**TL;DR**

従来のあらゆる認証フローには、避けられない瞬間がある。秘密がネットワークを渡る瞬間だ。Bearerトークン、リフレッシュトークン、ハッシュ化されたパスワード――サーバーは何かを受け取り、保存し、守り続けなければならない。Sealプルーフ認証はその前提を崩す。鍵はブラウザを出ない。サーバーが受け取るのは鍵の所持を証明するゼロ知識証明だ。そして現行の回路はさらに踏み込んだ: 鍵のハッシュさえ、ネットワークには出ない。

---

## 非技術者がまず抱く疑問

*「鍵を送らないのに、どうしてサーバーは本人だと分かるのか？」*

この疑問の直感は正しい。私たちは「認証とはサーバーが識別できる何かを送ること」だと学んできた。パスワード、トークン、クッキー。サーバーは受け取ったものと保存済みのものを照合する。

Sealプルーフ認証は仕組みが違う。ブラウザが鍵に対してGroth16ゼロ知識回路を実行し、*証明(proof)* と短い*ヌリファイア(nullifier)* を生成する。ヌリファイアは「この鍵」と「このセッション」に固有の単一フィールド要素だが、鍵そのものも鍵のハッシュも何も明かさない。サーバーは証明を検証し、登録済みのkey_hashを順にスキャンしてどれが今のセッションノンスに対して同じヌリファイアを生成するかを調べ、一致したらサインイン完了とする。

証明とヌリファイアがサーバーに届く。鍵は届かない。鍵のハッシュも届かない。

## ブラウザの中で何が起きているか

Lemma DashboardでSealプルーフサインインを選ぶと、3つのことが順に起きる。

**1. チャレンジ取得。** サーバーがランダムなノンスを発行し、有効期限5分の署名付きトークンに包む。このノンスが証明を「この特定のサインイン試行」に結びつけ、ヌリファイアをこのセッション限定のものにする。

**2. 証明生成。** APIキーが512ビット(64 ASCIIバイト)に展開される。WebAssemblyでブラウザ内で動く`seal-identity-v2` Groth16回路が、そのビット列を*プライベート入力*として、ノンスを*パブリック入力*として受け取る。回路の内部では次のことが起きる:

- `keyHash = SHA-256(keyBits)` を計算する――ただし**中間シグナルとして扱い、公開出力には出さない**
- 256ビットのhashを2つの128ビットフィールド要素に分割: `keyHash_hi` と `keyHash_lo`
- `nullifier = Poseidon(keyHash_hi, keyHash_lo, nonce)` を計算する

証明が生成する公開シグナルはちょうど2値: `nullifier` と `nonce` だけだ。鍵ビットもSHA-256ハッシュも、いかなる出力にも現れない。

ブラウザで動くコードはそのままこれだ:

```ts
import { apiKeyToBits } from "@lemmaoracle/seal";
import { create, prover } from "@lemmaoracle/sdk";

const witness = {
  keyBits: apiKeyToBits(key).map(Number),
  nonce: challenge.nonce,
};

const client = create({});
const { proof, inputs } = await prover.prove(client, {
  circuitId: "seal-identity-v2",
  witness,
});
```

**3. 証明の送信。** 証明と2つの公開シグナル(`nullifier`、`nonce`)をサーバーへ送る。サーバーはLemma RelayでGroth16証明を検証し(snarkjsはCloudflare Workersで利用できないAPIに依存するため)、D1の全アクティブ`api_keys`行をスキャンして各`key_hash`について`Poseidon(keyHash_hi, keyHash_lo, nonce)`を計算し、送信されたヌリファイアと一致するものを探す。一致すればスコープとアカウントを特定し、セッションクッキーを発行する。

ネットワークログに残るのは、`/api/auth/seal`へのPOSTリクエスト1本。中身は`proof`、`publicSignals`(2値)、`token`。APIキーなし。keyHashなし。

## Bearerトークンやリフレッシュトークンとどこがどう違うか

**サーバーが保存するもの。** Bearerトークン方式ではサーバーが秘密そのもの(あるいは直接比較で検証できる形式)を保存する。リフレッシュトークン方式では長命なトークンをサーバー側で保持する。どちらもDB漏洩で認証情報が露出する。Sealプルーフ認証でサーバーが保存するのは`key_hash`だけだ。しかもそのハッシュは認証時にネットワークを渡らない。サインインリクエストを傍受した攻撃者が見るのはヌリファイアだけ――鍵について何も明かさず、再利用もできないセッション固有の値だ。

**リプレイ耐性。** 漏洩したBearerトークンは有効期限まで使える。漏洩したSeal証明は無効だ。ヌリファイアはこのセッションのノンスにのみバインドされている。同じ鍵が異なる2つのセッションで証明を生成すると、まったく無関係な2つのヌリファイアが生まれる。相関を取る手がかりがない。

**観察者が学べること。** Bearerトークンでは、認証リクエストを見た観察者はトークン――再利用可能な認証情報――を得る。Sealプルーフ認証では、観察者が得るのはバインド済みノンスに対してのみ有効なヌリファイアで、サーバーはそのノンスを再発行しない。証明トランスクリプトを丸ごと取得しても、実行可能なものは何も得られない。

**より本質的な違い。** Bearerトークンフローでは信頼は*秘密の所持*によって確立される。Sealプルーフフローでは信頼は*ハッシュ原像の知識の証明*によって確立される――しかもそのハッシュ自体もPoseidonコミットメントの後ろに隠されている。秘密は証明回路の中に参加するが出力には現れない。ハッシュはヌリファイア導出に参加するが、こちらも出力には現れない。あなたの鍵もそのハッシュも見ることができないサーバーは、どちらも漏洩させることができない。

## ヌリファイアとサーバー側スキャン

新しい設計が生む問い: サーバーが`keyHash`を見られないなら、どのアカウントのものかをどうやって特定するのか？

答えは線形スキャンだ。サーバーは登録済みの各key_hashについて`Poseidon(keyHash_hi, keyHash_lo, nonce)`を計算し、送信されたヌリファイアと比較する。PoseidonはZKフレンドリーなハッシュ関数で、まさにこの用途向けに設計されている――JavaScriptで1件あたりサブミリ秒で動く(`poseidon-lite`: 純JS実装で、Cloudflare Workersと互換し、回路が使うcircomlibの定数と一致する)。現実的な登録API鍵数であれば、CloudflareワーカーのCPUバジェット内で十分完了する。

これがv2のプライバシーと計算量のトレードオフだ: サインイン1回あたり少し多く計算する代わりに、key_hashは回路の外には一切出なくなる。

## シンプルなユースケース

Lemma APIキーでユーザーが認証する開発者ツールを構築しているとする。バックエンドが生の鍵を受け取ったり保存したりせずにサインインをサポートしたい――さらに言えば、ハッシュさえも持ちたくない。ハッシュがあれば、セッションをまたいで同じユーザーを追跡できてしまうからだ。

Sealプルーフ認証では、証明送信のステップまでサインインフローは完全にクライアントサイドで完結する。ユーザーがローカルで鍵を入力し、回路がセッションノンスにバインドされたヌリファイアを生成し、バックエンドは証明を検証してハッシュスキャンでアカウントを解決する。各セッションが異なるヌリファイアを生成するため、自社のログからでも同じ鍵によるサインインをまたいで相関づけることはできない。

AIエージェントがセッションに自らサインインする場合も同じ原則が使える。エージェントが生の鍵を持ち、証明を生成する。オーケストレーター層はハッシュのみを保存し、ヌリファイアのみを受け取る。

## サーバーが触れないもの

サーバーは`proof`、2つの公開シグナル(`nullifier`、`nonce`)、署名付きチャレンジトークンを受け取る。証明が数学的に有効であることを検証する。チャレンジトークンのノンスを確認する。key_hashをスキャンして送信されたヌリファイアを生成するものを見つける。セッションを発行する。

この過程でサーバーがあなたのAPIキーもそのSHA-256ハッシュも参照、送受信、保存、照合することは一切ない。鍵はあなたのマシンに存在した。ハッシュは回路の中に生きた。証明がそれを証明した。ヌリファイアが確認した――このセッション一度だけ。
