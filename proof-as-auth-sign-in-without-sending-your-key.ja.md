**TL;DR**

従来のあらゆる認証フローには、避けられない瞬間がある。秘密がネットワークを渡る瞬間だ。Bearerトークン、リフレッシュトークン、ハッシュ化されたパスワード――サーバーは何かを受け取り、保存し、守り続けなければならない。Sealプルーフ認証はその前提を崩す。鍵はブラウザを出ない。サーバーが受け取るのは、鍵の所持を証明するゼロ知識証明だ。偽造不可能で、ノンスにバインドされ、リプレイもできない。

---

## 非技術者がまず抱く疑問

*「鍵を送らないのに、どうしてサーバーは本人だと分かるのか？」*

この疑問の直感は正しい。私たちは「認証とはサーバーが識別できる何かを送ること」だと学んできた。パスワード、トークン、クッキー。サーバーは受け取ったものと保存済みのものを照合する。

Sealプルーフ認証は仕組みが違う。ブラウザがあなたの鍵に対してGroth16ゼロ知識回路を実行し、*証明(proof)* を生成する。証明は短い数学的オブジェクトで、「特定のハッシュに一致する鍵の持ち主である」ということを、鍵そのものを明かさずに検証者へ伝える。そのハッシュはAPIキー作成時に登録済みだ。サーバーが保存するのはハッシュだけで、生の秘密は一切ない。

証明がサーバーに届く。鍵は届かない。

---

## ブラウザの中で何が起きているか

Lemma DashboardでSealプルーフサインインを選ぶと、3つのことが順に起きる。

**1. チャレンジ取得。** サーバーがランダムなノンスを発行し、有効期限5分の署名付きトークンに包む。このノンスが証明を「この特定のサインイン試行」に結びつける。

**2. 証明生成。** APIキーが256ビットに展開される。そのビット列と先ほどのノンスが、WebAssemblyでブラウザ内で動く`seal-identity-v1` Groth16回路の*プライベート入力*(witness)として渡される。回路が証明と公開シグナルを生成する。公開シグナルは、鍵ビットのSHA-256ハッシュ(256シグナル)とノンス(1シグナル)だ。鍵ビットはwitnessとして回路に消費され、出力には現れない。

ブラウザで動くコードはそのままこれだ:

```ts
import { apiKeyToBits } from "@lemmaoracle/seal";
import { create, prover } from "@lemmaoracle/sdk";

const witness = {
  keyBits: apiKeyToBits(key).map(Number),
  nonce: challenge.nonce,
};

const client = create({});
const { proof, inputs } = await prover.prove(client, {
  circuitId: "seal-identity-v1",
  witness,
});
```

**3. 証明の送信。** 証明と公開シグナルをサーバーへ送る。サーバーはGroth16証明を検証し(snarkjsはCloudflare Workersで利用できないAPIに依存するため、Lemma Relayに委譲)、公開シグナルの先頭256個から`keyHash`を読み出す。そのハッシュを`api_keys`テーブルで照合し、対応するスコープとアカウントを特定し、セッションクッキーを発行する。

ネットワークログに残るのは、`/api/auth/seal`へのPOSTリクエスト1本だ。中身は`proof`、`publicSignals`、`token`。APIキーはない。生の秘密は一切含まれない。

## Bearerトークンやリフレッシュトークンとどこがどう違うか

従来のトークンベース認証にはよく理解された脅威モデルがある。Sealプルーフ認証が違うのは、ポリシーの違いではなく仕組みの違いだ。

**サーバーが保存するもの。** Bearerトークン方式では、サーバーは秘密そのもの――あるいは直接比較で検証できる形式のもの――を保存する。リフレッシュトークン方式では長命なトークンをサーバー側で保持し、短命なアクセストークンを発行する。どちらも、データベースが漏洩すれば認証情報が露出する。Sealプルーフ認証でサーバーが保存するのは`key_hash`だけだ。ハッシュを知っても認証はできない。攻撃者は有効なZK証明を生成する必要があり、それには原像(鍵)を知っていなければならない。

**リプレイ耐性。** 漏洩したBearerトークンは有効期限まで使える。漏洩したSeal証明は無効だ。すべての証明はその特定のサインイン試行に対して発行されたチャレンジノンスにバインドされている。同じ証明を再送しても即失敗する。サーバーは証明内のノンスと自分が発行したチャレンジトークンを照合し、それは一度しか使えない。

**サーバーが守るべき表面積。** Bearer/リフレッシュトークン方式では、トークンストア――クエリ可能な秘密データベース――を守り続けなければならない。Sealプルーフ認証で守るべきは`key_hash`カラムだけだ。ハッシュは公開入力グレードのデータで、公開されても認証には使えない。鍵が漏洩した場合は当然ローテーションが必要だが、ハッシュDBが漏洩してもアカウントは守られる。

**より本質的な違い。** Bearerトークンフローでは信頼は*秘密の所持*によって確立される。Sealプルーフフローでは信頼は*ハッシュ原像の知識の証明*によって確立される。これは数学的な保証の種類が異なる。秘密は証明回路の中に参加するが、出力には現れない。あなたの鍵を見ることができないサーバーは、それを漏洩させることもできない。

## シンプルなユースケース

Lemma APIの上に独自ダッシュボードを構築しているとする。ユーザーに自分のAPIキー――課金やデータアクセスも制御する鍵――を直接扱わせずにサインインさせたい。

Sealプルーフ認証では、証明送信のステップまでサインインフローは完全にクライアントサイドで完結する。鍵を受け取り、ブラウザで証明を生成し、証明だけをバックエンドへ送るフォームを作ればよい。バックエンドは登録済みハッシュと照合して証明を検証する。生の鍵はユーザーのマシンにとどまる。

エージェントにも同じパターンが使える。APIキーを持つAIエージェントは、Sealプルーフを生成することでセッションに自らサインインできる。オーケストレーター側のインフラは生の鍵を保存する必要がなくなる。登録時にハッシュを保存し、以降は証明だけを受け付ける。

## Dashboardについて

Lemma Dashboardは[dashboard.lemma.workers.dev](https://dashboard.lemma.workers.dev/signin)で静かに公開中だ。Sealプルーフサインインはセカンダリパスとして実装されている。初回サインインはGitHub OAuthが必要で、アカウントの作成と最初のAPIキーの発行を行う。一度鍵を持てば、その後のサインインはSealプルーフで完結できる。上のスニペットはサインイン画面そのままのコードで、Dashboardは「あなたのブラウザで動くもの」のプレビューとして常に表示している。

## サーバーが触れないもの

最初の疑問に答えて締める。

サーバーは`proof`、`publicSignals`(keyHash + nonce)、署名付きチャレンジトークンを受け取る。証明が数学的に有効であることを検証する。証明内のノンスが自分が発行したチャレンジと一致することを確認する。keyHashを記録と照合する。セッションを発行する。

この過程でサーバーがあなたのAPIキーを参照、送受信、保存、照合することは一切ない。鍵はあなたのマシンに存在した。証明がそれを証明した。それがすべての保証だ。
